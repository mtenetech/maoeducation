## Requisitos funcionales principales

### 1. Gestión académica por niveles
- El sistema debe permitir precargar y administrar niveles/cursos desde:
  - 1ro de básica
  - 2do de básica
  - ...
  - hasta 9no de básica
- Pero debe ser extensible para poder agregar más cursos/niveles en el futuro sin romper el sistema.
- No quiero lógica quemada únicamente para esos grados; quiero un diseño parametrizable.

### 2. Roles del sistema
Debe existir un sistema robusto de roles y permisos. Los roles iniciales serán:
- Admin
- Inspector
- Profesor
- Alumno / Padre de familia

Debes diseñar el sistema para que en el futuro sea fácil agregar más roles o permisos más granulares.

### 3. Funcionalidades por rol

#### Admin
- administrar usuarios
- administrar roles y permisos
- administrar cursos, paralelos, materias y periodos académicos
- parametrizar cómo funciona el sistema
- administrar catálogos configurables del sistema
- generar reportes
- configurar plantillas genéricas de certificados o libretas

#### Inspector
- registrar incidencias disciplinarias
- justificar faltas
- consultar historial de asistencia e incidencias
- dar seguimiento a novedades de alumnos

#### Profesor
- registrar faltas/asistencia
- registrar notas
- crear actividades académicas evaluables
- seleccionar el tipo de actividad desde un catálogo configurable
- subir notas de cada actividad
- cargar o administrar horario de clases según el alcance
- visualizar sus cursos, materias, estudiantes y calendario

#### Alumno / Padre de familia
- ver tareas y demás actividades enviadas
- ver notas
- ver asistencia/faltas
- ver justificaciones
- acceder a mensajería
- ver calendario/horario de clases

### 4. Sistema genérico de tipos de actividad
Este punto es muy importante.

No quiero que el sistema quede hardcodeado solamente para “tareas” y “lecciones”.
Quiero que exista una estructura genérica basada en:

- tipos de actividad
- actividades creadas a partir de esos tipos
- agrupación flexible de actividades en insumos o categorías de evaluación

Ejemplos de tipos de actividad:
- tarea
- lección
- prueba
- examen
- proyecto
- participación
- refuerzo
- otro

Requisitos:
- los tipos de actividad deben poder configurarse
- el administrador debe poder crear, editar, activar o desactivar tipos
- el profesor debe poder seleccionar un tipo al momento de crear una actividad
- una actividad debe heredar comportamiento base común, pero permitir metadatos o configuraciones futuras según el tipo
- no quiero lógica rígida ni hardcoded

### 5. Sistema de insumos académicos
El sistema debe permitir que las actividades creadas por el profesor se agrupen dentro de “insumos” o categorías de evaluación, por ejemplo:
- tarea 1 + tarea 2 = insumo 1
- lección 1 = insumo 2
- pruebas = insumo 3

Quiero que el diseño sea flexible:
- que los insumos puedan configurarse
- que el profesor o el administrador pueda decidir cómo agrupar actividades
- que no quede rígido ni hardcoded
- que la estructura permita en el futuro cambiar cómo se calculan los promedios

Por ahora no quiero que definas reglas finales complejas de promedio, pero sí quiero que diseñes el sistema de forma genérica y extensible para soportarlas luego.

### 6. Parametrización del sistema académico
El sistema debe ser parametrizable para definir cómo funciona académicamente:
- si trabaja por trimestres
- o por quimestres
- y potencialmente otros esquemas en el futuro

Por defecto, el sistema debe arrancar funcionando por trimestres.

Quiero una solución donde esta parametrización esté bien modelada y no obligue a reescribir lógica de negocio cuando cambie el esquema.

### 7. Mensajería
Debe existir un módulo de mensajería interna entre actores del sistema.
Por ejemplo:
- profesor ↔ padre/alumno
- inspector ↔ padre/alumno
- admin ↔ usuarios

No hace falta que sea chat en tiempo real complejo al inicio, pero sí debe quedar preparada una base sólida para evolucionarlo.

### 8. Calendario y horario de clases
- Debe existir una vista de calendario u horario
- El horario de clases podrá ser subido o gestionado por cada profesor
- Padres/alumnos deben poder visualizarlo
- Debe ser modelado correctamente para crecer luego

### 9. Reportería y certificados
- Debe existir capacidad de generar reportes
- Debe existir generación de certificados o reporte de calificaciones con un formato genérico inicialmente
- Quiero que el diseño contemple plantillas o mecanismos para modificar esos formatos en el futuro sin rehacer todo

### 10. Notificaciones y correos
- El sistema debería quedar preparado para enviar correos y notificaciones
- No es necesario implementarlo completamente ahora, pero sí dejar una arquitectura preparada para agregarlo luego de forma limpia
